## 4. Reader problem

The core problem the book addresses:

- **AI tools are powerful but forget project context.** Every new session starts cold; bigger context windows don't fix the underlying memory gap.
- **Chat history is not a reliable business memory.** It's hard to search, easy to lose, locked to one vendor, and not structured for reuse.
- **Normal notes are not structured enough for AI agents.** They optimize for human browsing, not for an agent that needs to know *what each item is* and *where to start*.
- **Project experience is scattered.** It lives across conversations, files, decisions, incidents, and daily work — never in one readable place.
- **Most people have no reusable system** for turning lived work experience into AI-readable context. They redo the same explaining, project after project.

The missing piece is not a smarter model — it's a stable external memory the user controls and any model can read.

---

## 5. Book solution

The book teaches a complete, practical system:

- **Local-first Markdown structure** — source of truth on your machine, in plain text any tool can read.
- **The Semantic OS Method** — organize by *meaning and role*, not by app or topic, so humans and machines can both reason about it.
- **A starter kit** — a copy-ready folder structure (Inbox, Distill, Knowledge, Index, Projects, AI System).
- **Project Maps** — a living front door per project: goal, status, decisions, open questions, next actions, reusable knowledge.
- **Decision Records** — capture *why* a choice was made so it isn't relitigated.
- **Incident Records** — learn from what went wrong, with root cause and prevention.
- **Action Feedback** — record what actually happened after acting, closing the loop.
- **Context Packs** — focused bundles handed to an AI for one task, keeping answers grounded and within limits.
- **AI collaboration rules** — system rules and guardrails that make the OS work with any model.
- **A GitHub companion repository** — starter kit, templates, and safe examples readers can copy immediately.

## 14. Quality rules for manuscript writing

Rules for future drafting (and for AI collaborators):

- **Every chapter must solve a real reader problem** — state the problem up front.
- **Use concrete examples** — show actual files, structures, and steps.
- **Avoid vague AI hype** — no miracle claims, no "the future is here" filler.
- **Avoid private customer details** — public-safe, fictional examples only.
- **Separate public examples from private experience** — lessons may inform the writing; specifics must be anonymized or invented.
- **Explain concepts before naming them** — define the idea, then introduce the term.
- **Prefer practical steps over theory** — theory only as much as needed to act.
- **Include checklists where useful** — make chapters actionable.
- **Keep terminology consistent** — one name per concept across the whole book (and matching the repo).
